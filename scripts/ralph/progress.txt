## Codebase Patterns
- All types go in `src/types/index.ts` ‚Äî no barrel exports, import directly
- New nullable DB fields added to existing interfaces should use `?:` (optional) to avoid breaking mock data and existing code
- Mock data in `src/data/mock.ts` uses the `Pedido` type ‚Äî any required field addition will break it
- DB column names use snake_case; TypeScript interface fields for DB-mapped types also use snake_case
- Existing separation columns in DB were renamed by US-002: `separado_por` ‚Üí `separacao_operador_id`, `separado_em` ‚Üí `separacao_iniciada_em`, `embalado_em` ‚Üí `embalagem_concluida_em`, `agrupamento_tiny_id` ‚Üí `agrupamento_expedicao_id`
- When renaming DB columns, update ALL source files that reference them (grep for old name in src/ ‚Äî includes API routes, components, mock data, lib services)
- PL/pgSQL functions referencing renamed columns must also be updated via CREATE OR REPLACE in the same migration
- Status values changed: `pendente` ‚Üí `aguardando_separacao`, `expedido` removed (mapped to `embalado`). Valid values: aguardando_nf, aguardando_separacao, em_separacao, separado, embalado, cancelado
- `siso_pedidos.id` is `text` (not uuid) ‚Äî all PL/pgSQL and API functions accepting pedido IDs must use `text[]`/`text`
- `siso_pedido_itens` has no `unidade` column ‚Äî default to 'UN' wherever unidade is needed
- Localizacao is in `siso_pedido_item_estoques`, joined via pedido_id + produto_id + empresa_id (for origin empresa)
- Filter by galpao via `empresa_origem_id` ‚Üí `siso_empresas.galpao_id` join (more robust than `separacao_galpao_id`)
- Order date column in DB is `data` (not `data_pedido`)
- TABLE-returning PL/pgSQL functions return 0 rows for "not found" ‚Äî callers check `.length === 0` or empty result
- GET /api/separacao returns `{ counts: SeparacaoCounts, pedidos: array }` ‚Äî not a raw array
- Role-based filtering in separacao APIs uses `X-User-Cargo` header (not session-based galpao_id)
- Supabase FK join syntax: `siso_empresas(nome)` in select ‚Äî returns `{ nome }` object on each row
- UF/cidade are NOT stored in siso_pedidos ‚Äî return null (payload_original only stores Tiny order ID)

## 2026-03-10 - US-004
- Created PL/pgSQL function `siso_processar_bip_embalagem(p_sku text, p_galpao_id uuid, p_quantidade int)`
- Finds oldest `separado`-status order in galpao by joining `empresa_origem_id` ‚Üí `siso_empresas.galpao_id`
- Matches item by SKU or GTIN, locks with `FOR UPDATE OF i`
- Increments `quantidade_bipada`, sets `bipado_completo` when `>= quantidade_pedida`
- Checks all items complete ‚Üí transitions pedido to `embalado` + sets `embalagem_concluida_em`
- Returns TABLE(pedido_id text, produto_id text, quantidade_bipada numeric, bipado_completo boolean, pedido_completo boolean)
- Returns empty result set (no rows) when no matching item found
- Applied migration via Supabase MCP
- Files changed: supabase/migrations/20260311_fn_processar_bip_embalagem.sql
- **Learnings for future iterations:**
  - Use `empresa_origem_id` ‚Üí `siso_empresas.galpao_id` join to filter by galpao (more robust than `separacao_galpao_id` which may not be set)
  - Order date column is `data` (not `data_pedido`) in `siso_pedidos`
  - `FOR UPDATE OF i` locks just the item row in a joined query ‚Äî specify the table alias
  - TABLE-returning functions return empty result set (0 rows) for "not found" ‚Äî callers check for empty result, not NULL
  - When calling via Supabase RPC: `supabase.rpc('siso_processar_bip_embalagem', { p_sku, p_galpao_id, p_quantidade })`
