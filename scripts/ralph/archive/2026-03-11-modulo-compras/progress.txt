## Codebase Patterns
- All types go in `src/types/index.ts` ‚Äî no barrel exports, import directly
- PedidoItem interface maps 1:1 to siso_pedido_itens columns (snake_case)
- Domain types use Portuguese names, technical types use English
- Typecheck: `npx tsc --noEmit` (silent output = pass)
- `siso_pedido_itens.id` is `bigint` (not uuid) ‚Äî string representation in TypeScript
- `fornecedor_oc` already existed on siso_pedido_itens ‚Äî always verify columns before migrations
- `status_separacao` CHECK constraint name: `siso_pedidos_status_separacao_check`
- Supabase apply_migration is transactional ‚Äî all-or-nothing
- `ProcessedItem` fields in webhook-processor.ts are spread directly into siso_pedido_itens upsert ‚Äî add a field there to auto-persist to DB
- `fornecedor_oc` is already populated during webhook enrichment via `getFornecedorBySku()`
- Supabase `.select()` with FK join uses `table_name(columns)` syntax ‚Äî e.g. `siso_pedidos(numero)`
- GET API routes in this codebase don't enforce server-side auth (PIN auth is client-side only). Add `cargo` query param for lightweight checks.
- `siso_fila_execucao` insert requires: pedido_id, tipo, empresa_id, decisao (filial_execucao/operador are optional)
- Use `{ count: "exact", head: true }` for count-only queries in Supabase
- Supabase `.single()` returns error code `PGRST116` when no rows found ‚Äî use for 404 detection
- Next.js 16 dynamic route params are a Promise ‚Äî `const { param } = await params`
- AppShell handles login redirect + loading state; for custom cargo checks, add logic inside AppShell children
- GET /api/compras returns `{ counts, data }` ‚Äî counts has all 3 tab badge numbers regardless of requested status
- Supabase nested FK joins work 2+ levels deep: `table(col, nested_table(col))` ‚Äî cast result with `as unknown as T`
- `siso_pedido_item_estoques` numeric columns need `Number()` conversion for JS arithmetic
