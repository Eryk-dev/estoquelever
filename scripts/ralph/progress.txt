## Codebase Patterns
- All types go in `src/types/index.ts` — no barrel exports, import directly
- New nullable DB fields added to existing interfaces should use `?:` (optional) to avoid breaking mock data and existing code
- Mock data in `src/data/mock.ts` uses the `Pedido` type — any required field addition will break it
- DB column names use snake_case; TypeScript interface fields for DB-mapped types also use snake_case
- Existing separation columns in DB were renamed by US-002: `separado_por` → `separacao_operador_id`, `separado_em` → `separacao_iniciada_em`, `embalado_em` → `embalagem_concluida_em`, `agrupamento_tiny_id` → `agrupamento_expedicao_id`
- When renaming DB columns, update ALL source files that reference them (grep for old name in src/ — includes API routes, components, mock data, lib services)
- PL/pgSQL functions referencing renamed columns must also be updated via CREATE OR REPLACE in the same migration
- Status values changed: `pendente` → `aguardando_separacao`, `expedido` removed (mapped to `embalado`). Valid values: aguardando_nf, aguardando_separacao, em_separacao, separado, embalado, cancelado
