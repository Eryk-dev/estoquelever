## Codebase Patterns
- Tiny API functions use `tinyFetch<T>` helper with rate limiting and retry on 429
- API response types are exported interfaces prefixed with `Tiny`
- Tiny API v3 `POST /expedicao` accepts `idsPedidos` (order IDs) not just `idsNotasFiscais`
- `ObterEtiquetasResponseModel` returns `{ urls: string[] }`
- Always check `api tiny.json` in project root for endpoint schemas
- PrintNode API key from `process.env.PRINTNODE_API_KEY`, printer resolved via `resolverImpressora(usuarioId, galpaoId)`
- pedido.id is UUID, pedido.numero is the Tiny numeric ID — use numero for Tiny API calls
- etiqueta_url contains personal data (LGPD) — never expose to frontend
- Fire-and-forget pattern: call async fn (no await), `.catch()` for logging, return response after — used in bipar and aprovar endpoints
- Session validation: `getSessionUser(request)` from `src/lib/session.ts` — returns `SessionUser | null`
- Admin bypass for galpão restrictions: check `session.cargo !== "admin"` before galpão comparison
- Reprint pattern: try cached URL → refetch via agrupamento → full flow fallback
- etiqueta_status values: "pendente" | "imprimindo" | "impresso" | "falhou" (masculine, not "impressa")
- Separacao frontend components: `src/components/separacao/` — receive data as props, parent handles fetching/polling
